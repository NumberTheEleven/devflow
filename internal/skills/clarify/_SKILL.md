# /devflow:clarify — 需求澄清

## 何时使用

你有一个粗略的想法或模糊的需求。它可能来自：
- 你在 `/devflow` 流程的项目发现阶段选择的一项优化项
- 一个你想探索的功能想法
- 一个需要进一步细化的问题陈述

## 流程

### 步骤 1：复述想法

用 1-2 句话复述你的理解。在继续之前，请用户确认。

### 步骤 2：通过对话探索

每次只问一个问题。每个问题都应加深理解：

**需要探索的关键维度：**
- **目的：** 这解决了什么问题？谁受益？
- **成熟度预期：** 这是一个可丢弃的原型、用于验证的 MVP，还是一个生产级功能？期望的打磨程度是什么？AI 默认生成演示级代码 —— 如果不明确这一点，会导致代码看起来已完成，但尚未准备好投入实际使用。
- **约束条件：** 有任何技术、时间或资源约束吗？
- **成功标准：** 我们如何知道这件事已经完成且做得很好？包括功能性的（它能工作）和非功能性的（它能处理错误、覆盖边界情况、具备完整的 UI 状态）。
- **范围：** 什么在范围内？什么明确不在范围内？
- **用户：** 最终用户是谁？他们的工作流是什么？
- **现有代码与系统现状：** 代码库中是否有需要关注的相关部分？当前页面或功能长什么样？**这类问题属于"现状查证"，必须由 AI 自行完成，不得向用户提问。**

**对话规则：**
- 每轮只问一个问题
- 当选项明确时，优先使用选择题
- 探索阶段使用开放式问题也可以
- 问了 3-4 个问题后，总结已了解的内容并请用户确认
- **严禁向用户提出"现状查证"类问题。** 凡是可以通过查看代码、读取配置文件或截取页面截图得到答案的问题，都应由 AI 自行查证。例如："当前登录页面有没有验证码？"、"这个 API 返回什么字段？"、"用户设置页面在哪里？"都属于现状问题，不应提问。

## 现状查证规则（必读）

在需求澄清阶段，**只有"需求"类问题才允许向用户提问；"现状"类问题必须自行查证。**

### 区分"需求"与"现状"

| 类型 | 例子 | 处理方式 |
|------|------|----------|
| **需求** | "你希望用户如何完成登录？"、"这个功能的优先级是什么？"、"出现错误时应该展示什么文案？" | 可以向用户提问 |
| **现状** | "当前登录页面有没有验证码？"、"这个 API 返回什么字段？"、"用户设置页面在哪里？"、"当前页面是怎么布局的？" | **AI 自行查证，禁止提问** |

### 查证手段

遇到现状类问题时，按以下顺序自行验证：

1. **代码与配置搜索**
   - 使用 `Glob`、`Grep` 查找相关文件、组件、路由、API
   - 使用 `Read` 读取关键文件内容
   - 使用 `Bash` 运行项目命令查看输出（如 `npm run dev` 的启动日志）

2. **页面截图（如项目可运行）**
   - 使用 Playwright 工具导航到相关页面并截图
   - 例如：`browser_navigate` → `browser_take_screenshot`
   - 对 UI/UX 现状问题优先使用截图，避免让用户描述界面
   - **截图保存路径：** 统一保存到 `devflow/<feature>/screenshots/clarify/`，文件名如 `clarify-<page-name>-<YYYY-MM-DD>-<HHMMSS>.png`。在最终需求总结中引用该路径。

3. **运行服务查看行为**
   - 如项目已启动，通过 Playwright 执行真实操作（点击、输入）观察当前行为
   - 不要问用户"现在点击会发生什么"，而是自己去试

4. **记录假设**
   - 如果确实无法自行查证（如需要私有账号、第三方系统权限、未部署环境），将问题标记为"待澄清项"，在 Phase 2 拆解时再次确认
   - 在 clarify 阶段，不得以"请问现状是..."的形式向用户提问，只允许在总结中说明"我已自行查证到的现状是..."

### 澄清总结中的现状陈述

在最终的需求澄清总结中，除了目标、范围、成功标准等需求内容外，还应包含：

> **已查证的现状：**
> - 相关文件：`src/pages/Login.tsx`、`src/api/auth.ts`
> - 当前页面截图：...（如适用）
> - 当前行为：...

这些现状陈述用于支持后续方案设计，但获取它们的过程不应依赖用户回答。

### 步骤 3：提出 2-3 种方案

一旦需求足够清晰，提出 2-3 种方案：

```markdown
## 方案

### 方案 A：[名称]
**做法：** [1-2 句话]
**优点：** ...
**缺点：** ...

### 方案 B：[名称]
**做法：** [1-2 句话]
**优点：** ...
**缺点：** ...

### 推荐
[推荐哪个方案及原因]
```

优先展示推荐的方案并解释原因。

### 步骤 4：收敛为明确需求

用户选定方案后，生成一份清晰的需求摘要：

```markdown
## 已澄清的需求

**目标：** [一句话]

**做什么：**
- 需求 1
- 需求 2
- ...

**不做什么：**
- 非目标 1
- 非目标 2

**成功标准：**
- 标准 1
- 标准 2

**约束条件：**
- 约束 1
```

## 交接

一旦需求清晰且用户确认后，建议继续主流程：
- `/devflow` — 进入需求拆解及后续阶段

不要自动进入下一阶段。等待用户确认。
