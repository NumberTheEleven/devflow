四、DevFlow 的独特定位

三大差异化优势
1. 全链路编号追踪
R-001 → T-001 → TC-001 三条编号链路贯穿需求→任务→测试， 每个环节的状态可追踪。Superpowers/OpenSpec 缺失此能力。
2. 智能回退 + 增量重做
用户只需说"需求有问题"，AI 自动识别归属阶段， 只重做受影响的增量，已验证的全跳过。业界唯一。
3. 三态需求入口
/discover (无想法) → /clarify (模糊) → /breakdown (清晰)。 覆盖从零到一的完整需求光谱，Superpowers/OpenSpec 只覆盖后两种。
与行业共识的对齐
✅ 对齐的部分:
· 流程干线 = 五大公司一致模式 (需求→设计→实施→测试)
· PRD Before Design (Google 标准，业务流程图优先于规格)
· 方案对比+Alternatives (Google Design Doc 核心要求)
· Spec 含 Non-Goals 边界 (Google/Bean-to-Cup 共识)
· Sub-agent 并行 (Alibaba R2C 嵌套Agent 模式)
· TDD + 代码审查 (Superpowers/字节跳动 质量门禁)

🆕 DevFlow 独有的增量:
· 三态需求入口 → 任何起点的用户都能进入流程
· 编号追踪贯穿全链路 → 需求到测试可追溯
· 智能回退 → 减少返工浪费
· 非技术人员流程图 → PM/业务方可参与确认

五、DevFlow 最终命令流程

Phase 1 — Requirements 需求管理
/devflow:discover 无想法 → 扫描项目 → 优化机会列表
/devflow:clarify 模糊需求 → 逐轮对话 → 清晰描述
/devflow:breakdown 清晰需求 → 编号清单 (R-001...) 📄

Phase 2 — Design 方案设计
/devflow:blueprint 需求清单 → 业务流程图 → Spec边界 → 测试用例 (TC-001...) 📄

Phase 3 — Implementation 编码实现
/devflow:implement 蓝图 → 任务拆解 (T-001...) 📄 → 技术架构图 → Sub-agent编码

Phase 4 — Verification 测试验证
/devflow:verify 代码完成 → 逐项验证 TC + R → 智能回退 ←→ 闭环
