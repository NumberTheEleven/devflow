最终命令设计 — 内部流程明确

6 个命令 · 每个命令的内部步骤基于产出物依赖关系排序

/discover — Project Discovery 项目探索
输入: 无（用户没有明确需求）
内部流程:
① 扫描项目结构与代码质量 → ② 识别可优化点（性能/架构/可维护性/功能缺口） → ③ 生成优先级建议列表
输出: 优化机会列表 → 用户选择一项 → 触发 /clarify
/clarify — Requirements Clarification 需求澄清
输入: 模糊需求 / 从 /discover 选中的优化点
内部流程:
① 逐轮对话展开：目的/约束/成功标准 → ② 提出 2-3 方案对比（trade-off + 推荐） → ③ 收敛为清晰需求描述
输出: 清晰的需求描述（What & Why） → 触发 /breakdown
/breakdown — Requirements Breakdown 需求拆解
输入: 清晰需求描述（或直接进入）
内部流程:
① 拆解为带编号的子需求清单 → ② 标注每个子需求的优先级( P0/P1/P2 )和依赖关系 → ③ 用户确认清单
输出: 编号需求清单（每个需求可追踪状态） → 触发 /blueprint
/blueprint — Solution Blueprint 方案蓝图
输入: 编号需求清单
内部流程 (基于依赖关系排序):
① 业务流程图 — 先确认用户世界里的业务流程（需求→方案的接口层）
② 规格边界 & 规范标准 — 基于流程图定义实现范围和技术规范
③ 测试用例清单 — 基于明确的范围，列出带编号追踪的测试用例
输出: 业务流程图 + Spec/Scope + 测试用例清单 → 触发 /implement
/implement — Implementation 编码实现
输入: 蓝图（Spec + 流程图 + 测试清单）
内部流程:
① 技术架构文档 — 选择合适的 UML/泳道图/状态机图描述技术方案
② 任务依赖分析 — 识别无依赖任务（可并行 Sub-agent）vs 有依赖任务（顺序 TDD）
③ 并行 Sub-agent 编码 — TDD + 代码审查 + 逐任务提交
输出: 完成编码 + 提交记录 → 触发 /verify
/verify — Verification 测试验证
输入: 完成编码
内部流程:
① 逐项跑测试用例清单（/blueprint 阶段定义的）
② 逐项核对需求 Checklist（/breakdown 阶段定义的）
③ 失败项 → 智能定位所属阶段 → 自动回退
④ 全绿 → 完成闭环，生成验证报告
输出: 验证报告 / 回退到对应阶段重做（增量）

设计依据 — 行业标准参考

Google Design Doc 标准: PRD (What/Why) → Design Doc (How) → Work Plan

字节跳动 5 步标准化: 流程标准化 → 质量门禁 → 需求研发双流联动 → 全流程洞察 → 一站式体验

本插件核心原则: 每个命令内部的产出物顺序 = 依赖关系决定 —— 被依赖的先产出，无依赖的可并行
