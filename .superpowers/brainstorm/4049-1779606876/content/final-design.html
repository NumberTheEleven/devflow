Phase 1: Requirements 需求管理

/discover
Project Discovery
触发场景: 用户无明确需求
流程:
① 扫描项目结构与代码
② 识别可优化点
③ 生成优先级建议列表
输出: 优化机会列表 → 触发 /clarify
/clarify
Requirements Clarification
触发场景: 用户有模糊想法
流程:
① 逐轮对话展开需求
② 2-3 方案对比 + 推荐
③ 收敛为清晰描述
输出: 清晰需求 → 触发 /breakdown
/breakdown
Requirements Breakdown
触发场景: 需求清晰（可直接进入）
流程:
① 拆解为编号子需求 R-001...
② 标注优先级 P0/P1/P2
③ 标注依赖关系
输出: 📄 需求清单（落盘）

Phase 2: Design 方案设计

/blueprint
Solution Blueprint
触发场景: 需求清单就绪（可直接进入）
流程 (按依赖关系排序):
① 业务流程图 — 非技术人员可读，先确认用户世界的流程
② 规格边界 & 规范标准 — 基于流程图定义实现范围
③ 测试用例清单 — TC-001... 带编号追踪
输出: 📄 设计规格文档 + 📄 测试用例清单（落盘）

Phase 3: Implementation 编码实现

/implement
Implementation
触发场景: 蓝图就绪（可直接进入）
流程:
① 任务拆解 — T-001, T-002... 标注影响模块/文件（📄 落盘）
② 技术架构文档 — UML/泳道图/状态机图
③ 编码执行 — 无依赖并行 Sub-agent + 有依赖顺序 TDD

Phase 4: Verification 测试验证

/verify
Verification
触发场景: 编码完成（可直接进入）
流程:
① 逐项验证 TC-xxx 测试用例
② 逐项核对 R-xxx 需求清单
③ 失败 → 智能定位归属阶段 → 自动回退
④ 全绿 → 验证报告 + 闭环

关键机制

灵活入口
任意阶段可直接进入。
例如：用户已有 PRD → 直接 /breakdown
用户已有设计文档 → 直接 /implement
智能回退
用户说"需求不对" → AI 自动回退到 /clarify
用户说"设计有问题" → AI 自动回退到 /blueprint
增量重做：只改受影响的链条
阶段收尾询问
每阶段完成时询问：
1. 进入下一阶段
2. 有疑问需回退
3. 暂不继续

落盘文件结构 (项目内)

devflow/
├── requirements.md   # R-001 ~ R-xxx 需求清单 + 状态
├── design.md        # 设计规格 + 业务流程图
├── tasks.md           # T-001 ~ T-xxx 任务拆解 + 状态
└── test-cases.md      # TC-001 ~ TC-xxx 测试用例 + 状态

技术方案

插件类型Claude Code Plugin（Skills + Commands）
文件格式每个命令 = 一个 Skill Markdown 文件
发布目标Claude Code Marketplace + Eleven Marketplace
命令调用/devflow:discover, /devflow:clarify, ...
状态持久化混合：核心清单落盘 devflow/*.md + 执行细节靠会话上下文
