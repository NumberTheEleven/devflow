## R-002: L2 交互验证层（新增核心）

**优先级:** P0
**状态:** 待开始
**描述:** 对 P0 页面执行关键操作链路，不是只看页面能否打开，而是真正模拟用户操作：点击主按钮 → 观察响应 → 填写表单 → 触发表单校验 → 检查反馈等。使用 Playwright MCP 执行操作，DOM snapshot 记录前后状态。Playwright 不可用时降级为手工交互验证（人操作 + 模型引导）。
**依赖:** R-001
**验收标准:**
- [ ] 从 `test-cases.md` 中识别需要交互验证的 TC
- [ ] 对每个交互 TC 自动执行操作步骤（navigate → click → type → wait → snapshot）
- [ ] 不依赖具体文案（如"提交成功"），用 DOM 结构变化判断操作是否生效
- [ ] Playwright MCP 不可用时，降级为引导用户手工执行并报告结果
- [ ] 每步操作产生 Interaction Trace（操作描述 + 前后 DOM diff）

## R-003: L3 结构化手工验证层（增强）

**优先级:** P0
**状态:** 待开始
**描述:** 对 L1/L2 无法自动覆盖的 TC（如"动画流畅"、"权限区分正确"、"多角色场景"），使用结构化的验证协议。每个 TC 必须有至少一种证据（DOM snapshot / console log / network log / user report）才能标记为"通过"，不允许无证据标记。
**依赖:** R-001, R-002
**验收标准:**
- [ ] 定义各 TC 类型的证据要求矩阵（什么类型 TC 需要什么证据）
- [ ] 无法自动验证的 TC 逐条向用户展示步骤、请求用户确认，并要求用户提供反馈
- [ ] 无证据的 TC 标记为 `unverified`，不许标记为 `done`
- [ ] 用户明确跳过的 TC 标记为 `skipped`，需附带跳过原因

## R-004: TC 智能路由引擎

**优先级:** P0
**状态:** 待开始
**描述:** 验证开始前，自动分析每个 TC 的内容和类型，判断适合 L1/L2/L3 哪一层，生成验证计划。避免所有 TC 都被路由到 L3 手工验证（太慢），也避免不适合自动化的 TC 被强行自动化（产生假阳性）。
**依赖:** R-001, R-002, R-003
**验收标准:**
- [ ] 按 TC 类型和内容关键词自动分类：`structural`（L1）/ `interaction`（L2）/ `manual`（L3）
- [ ] 路由结果展示给用户确认（允许用户调整）
- [ ] 分类规则清晰可查（定义在 verify skill 中）
- [ ] 路由决策本身记录到 `verification-log.md`

## R-005: 验证深度评分体系

**优先级:** P1
**状态:** 待开始
**描述:** 不再只看"X/Y 通过"，增加验证深度指标：多少 TC 被真正执行过操作步骤（而非仅静态检查），多少 TC 有证据支撑。暴露验证质量，避免"虚假全部通过"。
**依赖:** R-004
**验收标准:**
- [ ] 报告新增 "验证深度" 指标：被执行过的 TC 数 / 总 TC 数
- [ ] 报告新增 "证据覆盖度"：有证据的 TC 数 / 总 TC 数
- [ ] 深度 < 100% 时给出明确警告，不暗示验证完成
- [ ] 报告底部显示按 TC 类型的覆盖分布 (L1 / L2 / L3 各覆盖了多少)
