# 需求清单

> 生成时间: 2026-07-09
> 来源: /devflow — 需求拆解阶段

## R-001: 自动确认入口

**优先级:** P0
**状态:** 待开始
**描述:** 在现有 /devflow 各阶段 checkpoint 提供"自动确认/自循环"选项，允许用户在任意阶段结束时将后续流程交给 AI 自主推进。
**依赖:** 无
**验收标准:**
- [ ] 用户在 clarify/breakdown/blueprint/implement/verify 任一 checkpoint 可输入 `auto` / `autonomous` / `自循环` 等指令
- [ ] AI 识别指令后切换到自循环模式，接管后续阶段
- [ ] 默认流程（非自循环）保持原样，不受影响

---

## R-002: 自循环状态扩展

**优先级:** P0
**状态:** 待开始
**描述:** 扩展 `state.json`，持久化自循环运行所需的元数据与运行时状态。
**依赖:** R-001
**验收标准:**
- [ ] `state.json` 增加 `autonomous` 对象，至少包含：`enabled`、`started_at`、`resume_phase`、`started_from`、`current_loop`、`max_loops`、`timeout_at`、`last_report_at`、`status`（running/paused/failed/completed）
- [ ] 每次阶段推进、回退、迭代都更新 `state.json`
- [ ] 自循环中断后可通过 `/devflow` 恢复并继续

---

## R-003: AI 自评估标准

**优先级:** P0
**状态:** 待开始
**描述:** 定义各阶段产出的自检清单，使 AI 能在无用户确认时自行判定"阶段完成"。
**依赖:** R-001
**验收标准:**
- [ ] clarify：需求总结包含目标、包含项、排除项、成功标准、约束，无待澄清项
- [ ] breakdown：R-xxx 清单完整，优先级/依赖/验收标准清晰，无 open_questions
- [ ] blueprint：design.md + test-cases.md 已保存，流程图与 TC 覆盖所有 R-xxx
- [ ] implement：T-xxx 全部完成，tasks.md 已更新，代码已提交
- [ ] verify：verification-log.md 生成，L1/L2/L3 全部通过或已处理

---

## R-004: 阶段自动推进

**优先级:** P0
**状态:** 待开始
**描述:** 自循环模式下，AI 完成当前阶段自评后自动进入下一阶段，不再等待用户逐阶段确认。
**依赖:** R-002, R-003
**验收标准:**
- [ ] 自评通过后立即更新 `state.json` 并进入下一阶段
- [ ] 每个阶段开始时读取并遵循前置文档
- [ ] 推进前检查 `autonomous.timeout_at` 与 `max_loops`，超限则暂停报告

---

## R-005: 编码-验证自修复循环

**优先级:** P0
**状态:** 待开始
**描述:** implement 完成后进入 verify；验证失败时自动返回 implement 修复并重新验证。
**依赖:** R-004
**验收标准:**
- [ ] verify 发现 L1/L2 失败时，自动定位对应 T-xxx 并返回 implement
- [ ] 修复后重新运行 verify，直到通过或达到熔断条件
- [ ] 每次修复循环计数递增，写入 `state.json`

---

## R-006: 上游阶段自动回退

**优先级:** P0
**状态:** 待开始
**描述:** 当 bug/失败根因被判定为 design/requirements/clarify 时，自动回退到对应阶段重新梳理、修正、再推进。
**依赖:** R-005
**验收标准:**
- [ ] AI 能根据失败现象判断根因阶段（implement / blueprint / breakdown / clarify）
- [ ] 回退时更新 `rollback_history`，记录原因与目标阶段
- [ ] 回退后按正常流程重新完成该阶段，再继续下游

---

## R-007: 熔断与超时保护

**优先级:** P0
**状态:** 待开始
**描述:** 配置最大迭代次数与超时时间，防止自循环死循环或长时间空转。
**依赖:** R-002
**验收标准:**
- [ ] 默认最大循环次数（如 20）与超时时间（如 4 小时）可配置
- [ ] 达到限制时停止自循环，向用户报告当前阶段、已尝试次数、阻塞原因
- [ ] 用户回复后可选择继续、重置限制或退出自循环

---

## R-008: 关键节点报告

**优先级:** P0
**状态:** 待开始
**描述:** 仅在极少数真正需要人工介入时暂停并报告用户；常规推进不打扰用户。
**依赖:** R-001
**验收标准:**
- [ ] 遇到无法自愈的阻塞时暂停并报告
- [ ] 触及安全边界（如 force push、分叉、合并冲突）时暂停
- [ ] 需要用户做价值判断（如需求取舍、异常处理策略）时暂停
- [ ] 熔断触发时暂停
- [ ] 阶段正常完成、常规修复循环不发送报告
